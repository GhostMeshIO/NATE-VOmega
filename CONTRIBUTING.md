# Contributing to NATE vΞ⁷·₁

Thank you for your interest in contributing to the NATE framework. Please note that this is a **speculative theoretical framework**, not an active engineering project. Contributions are welcome in the following areas:

## Acceptable Contributions

1. **Mathematical corrections**: Identifying and fixing errors in the formalism
2. **Literature references**: Adding relevant peer-reviewed citations
3. **Simulation improvements**: Enhancing the in-silico models
4. **Documentation**: Clarifying explanations, adding examples
5. **Safety analysis**: Strengthening the safety framework
6. **Critical review**: Rigorous scientific critique of the framework's assumptions

## Contribution Guidelines

### Principle 1: Evidence-Based
All claims must be grounded in peer-reviewed literature or explicitly flagged as speculative. Do not add claims that imply empirical validation where none exists.

### Principle 2: Safety Non-Negotiable
Safety constraints are absolute and cannot be relaxed. Any suggestion to increase exposure limits, reduce safety margins, or bypass interlock systems will be rejected.

### Principle 3: Mathematical Rigor
All mathematical formalism must be internally consistent and physically plausible. Submissions that violate conservation laws or produce unphysical results will be rejected.

### Principle 4: Transparency
The speculative status of this framework must never be obscured. Language that implies engineering readiness, clinical applicability, or empirical validation will be edited.

### Principle 5: No Pseudoscience
Contributions that invoke supernatural, paranormal, or New Age concepts will be rejected. The framework's terminology should be grounded in established scientific disciplines.

## Pull Request Process

1. Fork the repository
2. Create a feature branch (`git checkout -b feature/your-contribution`)
3. Make your changes with clear commit messages
4. Submit a pull request with a description of the changes and their rationale
5. Respond to review comments within 30 days

## Code Style

- Python: Follow PEP 8
- JavaScript: Follow ESLint configuration
- Comments: Explain the physical/biological rationale, not just the code logic

## Reporting Issues

When reporting issues, please specify:
- The specific section or file where the issue occurs
- The nature of the issue (mathematical error, safety concern, missing citation, etc.)
- A proposed correction, if available
